1. LOS 3 CRUDs DEL PROYECTO


El proyecto tiene 3 integrantes, por lo tanto se presentan 3 CRUDs completos.
Ninguno es sobre la tabla de usuarios ni perfiles. Todos solo los puede acceder
el administrador.

Las rutas de los 3 CRUDs están definidas en:
  Archivo: routes/web.php (líneas 39–46)

  Route::middleware(['auth', 'role:admin'])->prefix('admin')->name('admin.')->group(function () {
      Route::resource('categories', CategoryController::class);
      Route::resource('dishes', DishController::class);
      Route::resource('restaurants', RestaurantController::class);
      ...
  });

  "Route::resource" genera automáticamente las 7 rutas del CRUD:
  index, create, store, show, edit, update, destroy.
  El middleware 'role:admin' bloquea el acceso a cualquier usuario que no sea admin.


----------------------------------------
CRUD 1: RESTAURANTES (Dark Kitchens)
----------------------------------------
Para este CRUD usamos el modelo Restaurant con Eloquent ORM, un controlador
con Route Model Binding y vistas Blade con Tailwind CSS. La ruta resource genera
las 7 acciones automáticamente. Se agregó un método show() adicional que sirve
como hub de gestión: desde ahí se añaden, editan y eliminan los platos del
restaurante sin salir de esa pantalla.

URL base:       /admin/restaurants
Controlador:    app/Http/Controllers/Admin/RestaurantController.php
Vistas:         resources/views/admin/restaurants/
                  - index.blade.php  (listado de restaurantes)
                  - create.blade.php (formulario nuevo)
                  - edit.blade.php   (formulario editar)
                  - show.blade.php   (gestión de platos del restaurante)
Modelo:         app/Models/Restaurant.php

¿Qué hace cada método del controlador?

  index()
    Obtiene todos los restaurantes con el conteo de platos asociados.
    Cada fila del listado tiene un botón "Gestionar" que lleva a show().
    Código usado: Restaurant::withCount('dishes')->latest()->get()
      withCount() agrega una columna virtual dishes_count sin cargar los registros.
      latest() ordena por created_at descendente.

  create()
    Devuelve la vista con el formulario vacío para crear un restaurante.
    Código usado: return view('admin.restaurants.create')

  store(Request $request)
    Valida nombre obligatorio, máximo 255 caracteres. Si falla la BD muestra error.
    Código usado: $request->validate([...]), Restaurant::create([...]),
      redirect()->route('admin.restaurants.index')->with('success', '...')
      El ->with() guarda el mensaje en sesión flash para mostrarlo en la vista.

  show(Restaurant $restaurant)
    Carga el restaurante con sus platos y la categoría de cada plato en una sola consulta.
    También carga las categorías activas para el selector del formulario de nuevo plato.
    Código usado: $restaurant->load(['dishes' => fn($q) => $q->with('category')->orderBy('name')])
      load() es lazy eager loading: carga relaciones sobre un modelo ya instanciado.
      La función anónima fn($q) permite aplicar condiciones dentro de la relación cargada.

  edit(Restaurant $restaurant)
    Laravel inyecta el modelo automáticamente buscando el ID en la URL (Route Model Binding).
    Devuelve la vista con los datos del restaurante ya cargados en el formulario.
    Código usado: return view('admin.restaurants.edit', compact('restaurant'))

  update(Request $request, Restaurant $restaurant)
    Valida los mismos campos que store y actualiza solo los campos enviados.
    Código usado: $restaurant->update([...])
      update() en Eloquent solo modifica los campos indicados, no toda la fila.

  destroy(Restaurant $restaurant)
    Verifica relaciones antes de intentar eliminar para evitar errores de BD.
    Código usado: $restaurant->dishes()->count() para la verificación previa,
      $restaurant->delete() para la eliminación,
      back()->with('error', '...') para redirigir con mensaje si no se puede eliminar.

¿Cómo se muestra el nombre y no el ID de la foreign key?
  En el listado de PLATOS, la columna "Restaurante" muestra:
    $dish->restaurant->name  (no el restaurant_id numérico).
  Esto funciona gracias a la relación definida en el modelo Dish:
    public function restaurant() { return $this->belongsTo(Restaurant::class); }


----------------------------------------
CRUD 2: PLATOS
----------------------------------------
Para este CRUD usamos el modelo Dish con dos foreign keys (restaurant_id y
category_id). El controlador usa eager loading con with() para evitar consultas
N+1. Se implementó creación de categoría al vuelo con firstOrCreate() para que
el admin pueda escribir una nueva categoría directamente en el formulario sin
tener que ir a otra pantalla. Las imágenes se gestionan con Storage de Laravel.

URL base:       /admin/dishes
Controlador:    app/Http/Controllers/Admin/DishController.php
Vistas:         resources/views/admin/dishes/
                  - index.blade.php  (listado con categoría y restaurante)
                  - create.blade.php (formulario nuevo)
                  - edit.blade.php   (formulario editar)
Modelo:         app/Models/Dish.php

¿Qué hace cada método del controlador?

  index()
    Carga todos los platos junto con su categoría y restaurante en una sola consulta.
    Código usado: Dish::with('category', 'restaurant')->orderBy('name')->get()
      with() hace eager loading: resuelve las relaciones con 2 queries extras en vez
      de ejecutar una query por cada plato (problema N+1).

  create()
    Carga categorías y restaurantes activos para poblar los selectores del formulario.
    Código usado: Category::where('active', true)->orderBy('name')->get()
      Se filtran solo los activos para no mostrar categorías deshabilitadas.

  store(Request $request)
    Valida los campos y crea el plato. La categoría puede venir del selector o como texto nuevo.
    Código usado: $request->validate([...]) para validar,
      Category::firstOrCreate(['name' => ...], ['active' => true]) para crear categoría
      al vuelo si se escribió una nueva (si ya existe con ese nombre, la reutiliza),
      $request->hasFile('image') y ->store('dishes', 'public') para guardar imagen,
      unset($data['new_category']) para limpiar el campo extra antes de Dish::create($data).

  edit(Dish $dish)
    Route Model Binding inyecta el plato. Carga selectores para el formulario de edición.
    Código usado: request('back') para leer el parámetro ?back={id} de la URL.
      La vista usa ese valor para mostrar el enlace "← Volver al restaurante" y para
      incluir el campo oculto back_to_restaurant en el formulario.

  update(Request $request, Dish $dish)
    Actualiza el plato. Si hay imagen nueva, borra la anterior para no acumular archivos.
    Código usado: Storage::disk('public')->delete($dish->image) para borrar la imagen
      anterior del servidor, ->store('dishes', 'public') para guardar la nueva,
      $dish->update($data) para actualizar solo los campos modificados.

  destroy(Dish $dish)
    Verifica que el plato no tenga pedidos antes de eliminarlo.
    Código usado: $dish->orderItems()->count() para la verificación previa,
      Storage::disk('public')->delete($dish->image) para borrar la imagen del disco,
      $dish->delete() para eliminar el registro,
      request()->filled('back_to_restaurant') para decidir a dónde redirigir.

¿Foreign keys con nombre, no con ID?
  En el listado de platos:
    $dish->category->name     (no category_id)
    $dish->restaurant->name   (no restaurant_id)
  En los formularios se muestran selectores con nombres, pero el valor
  enviado al servidor es el ID. Laravel valida que ese ID exista en la tabla.


----------------------------------------
CRUD 3: CATEGORÍAS
----------------------------------------
Para este CRUD usamos el modelo Category con validación de unicidad en el nombre
para evitar duplicados. El controlador protege el DELETE verificando si hay
platos asignados antes de intentar eliminar. Se usa withCount() para mostrar
cuántos platos tiene cada categoría sin hacer consultas adicionales.

URL base:       /admin/categories
Controlador:    app/Http/Controllers/Admin/CategoryController.php
Vistas:         resources/views/admin/categories/
                  - index.blade.php  (listado)
                  - create.blade.php (formulario nuevo)
                  - edit.blade.php   (formulario editar)
Modelo:         app/Models/Category.php

¿Qué hace cada método del controlador?

  index()
    Lista todas las categorías con el total de platos que tiene cada una.
    Código usado: Category::withCount('dishes')->orderBy('name')->get()
      withCount() agrega la columna dishes_count sin traer los platos completos.

  create()
    Devuelve la vista con el formulario vacío.
    Código usado: return view('admin.categories.create')

  store(Request $request)
    Valida que el nombre no se repita en toda la tabla y crea la categoría.
    Código usado: $request->validate(['name' => 'required|unique:categories', ...])
      La regla unique:categories compara contra todos los registros de esa tabla.
      $request->boolean('active', true) convierte el checkbox en true/false.
      Category::create($data) inserta el registro usando los campos del $fillable.

  edit(Category $category)
    Route Model Binding inyecta la categoría por su ID en la URL.
    Código usado: return view('admin.categories.edit', compact('category'))
      compact() crea el array ['category' => $category] de forma abreviada.

  update(Request $request, Category $category)
    Valida el nombre como único pero ignorando el registro que se está editando.
    Código usado: 'unique:categories,name,' . $category->id
      El tercer parámetro de unique le dice a Laravel qué ID ignorar en la comparación,
      así se puede guardar sin cambiar el nombre sin que falle la validación de unicidad.

  destroy(Category $category)
    Verifica que no haya platos antes de eliminar para proteger la integridad de la BD.
    Código usado: $category->dishes()->count() para la verificación previa,
      $category->delete() para eliminar el registro si pasa la verificación.

¿Se muestran IDs?
  No. En el listado de categorías solo se muestra nombre, descripción,
  cantidad de platos y estado. Los IDs autogenerados nunca aparecen en pantalla.


========================================
2. PROCESO SECUNDARIO
========================================

Para este proceso usamos el modelo Order con sus relaciones (client, items.dish),
dos rutas PATCH adicionales fuera del resource (una para estado del pedido y otra
para estado de pago Yape), y lógica condicional en el controlador que cambia
automáticamente el estado del pedido según la decisión del admin sobre el pago.
La vista diferencia pedidos Yape de tarjeta y bloquea cambios en pedidos finalizados.

El proceso secundario elegido es:
  "Gestión del estado de todos los pedidos por parte del Administrador"

URL:            /admin/orders
Controlador:    app/Http/Controllers/Admin/OrderController.php
Vista:          resources/views/admin/orders/index.blade.php

Flujo del proceso:

  1. El cliente hace un pedido desde /pedir/{restaurante}.
     Se crea el pedido con estado "pending" (pendiente) en la base de datos.

  2. El administrador entra a /admin/orders y ve todos los pedidos del sistema
     con su estado actual, cliente, platos y total.

  3. El administrador puede cambiar el estado de cualquier pedido usando el
     selector de la tabla y presionando "OK". Los estados posibles son:
       pending    → Pendiente
       confirmed  → Confirmado
       preparing  → Preparando
       ready      → Listo
       delivering → En camino
       delivered  → Entregado
       cancelled  → Cancelado

¿Cómo funciona el código del controlador?

  index()
    Carga todos los pedidos con sus relaciones en una sola operación.
    Código usado: Order::with('client', 'items.dish')->latest()->get()
      with('client') carga el usuario que hizo el pedido (FK client_id → users).
      with('items.dish') carga los ítems del pedido y el plato de cada ítem (relación anidada).
      latest() es equivalente a orderBy('created_at', 'desc').

  updateStatus(Request $request, Order $order)
    Valida que el nuevo estado sea uno de los 7 valores permitidos y lo guarda.
    Código usado: $request->validate(['status' => 'required|in:pending,confirmed,...'])
      La regla in: lista los valores exactos aceptados; cualquier otro valor falla.
      $order->update(['status' => $request->status]) actualiza solo ese campo.
      back()->with('success', '...') redirige a la página anterior con mensaje flash.

  updatePaymentStatus(Request $request, Order $order)
    Cambia el estado de pago Yape y ajusta automáticamente el estado del pedido.
    Código usado: $request->validate(['estado_pago' => 'required|in:Pendiente,Pagado,Rechazado'])
      match($request->estado_pago) { 'Pagado' => 'confirmed', 'Rechazado' => 'cancelled' }
      El match() de PHP 8 asigna el nuevo estado del pedido según la decisión del admin.
      $order->update([...]) actualiza estado_pago, fecha_pago y status en una sola llamada.
      Ruta: PATCH /admin/orders/{order}/payment-status (definida manualmente en web.php).

¿Los pedidos entregados/cancelados no se pueden cambiar?
  Sí. En la vista, los pedidos con estado "delivered" o "cancelled" muestran
  un guión "—" en vez del selector.

Escenarios de prueba del proceso secundario:

  Escenario A — Administrador gestiona pedido con Yape:
    1. Iniciar sesión con cliente@eathidden.com / password
    2. Ir a /pedir, elegir restaurante, agregar platos
    3. Seleccionar método de pago "Yape", escanear QR y confirmar pedido
    4. Cerrar sesión
    5. Iniciar sesión con admin@eathidden.com / password
    6. Ir a Administración > Pedidos (o /admin/orders)
    7. Ver el pedido con estado "Pendiente" y pago Yape "Pendiente"
    8. Confirmar el pago Yape → el pedido pasa automáticamente a "Confirmado"
    9. Continuar actualizando: "Preparando" → "Listo" → "En camino" → "Entregado"

  Escenario B — Administrador gestiona pedido con tarjeta:
    1. Iniciar sesión con cliente@eathidden.com / password
    2. Ir a /pedir, elegir restaurante, seleccionar método "Tarjeta"
    3. Ingresar datos de tarjeta y realizar pedido
    4. Iniciar sesión con admin@eathidden.com / password
    5. En /admin/orders cambiar el estado del pedido manualmente

  Escenario C — Pedido ya existe (datos del seeder):
    1. Iniciar sesión con admin@eathidden.com / password
    2. Ir a /admin/orders
    3. Ya aparece un pedido de prueba en estado "Preparando" (creado por el seeder)
    4. Cambiar su estado directamente desde la tabla


========================================
3. DISTRIBUCIÓN DEL CÓDIGO
========================================

El proyecto usa la arquitectura MVC de Laravel. Las rutas están agrupadas por rol
con middleware personalizado (CheckRole). Los modelos definen las relaciones entre
tablas con Eloquent. Los controladores están separados en una carpeta Admin/ para
los que solo usa el administrador. Las vistas usan un layout compartido (app.blade.php)
con sidebar dinámico según el rol del usuario autenticado.

El proyecto sigue la arquitectura MVC de Laravel (Modelo - Vista - Controlador).

  MODELOS (app/Models/)
    User.php         — Usuario con campo "role" (admin/kitchen/client/delivery)
    Restaurant.php   — Dark kitchen. Relación hasMany con Dish.
    Category.php     — Categoría de platos. Relación hasMany con Dish.
    Dish.php         — Plato. Relaciones belongsTo Restaurant y Category,
                       hasMany OrderItem.
    Order.php        — Pedido. Relaciones: belongsTo User (como cliente),
                       hasMany OrderItem. Campos de pago: metodo_pago
                       (Card/Yape), estado_pago (Pendiente/Pagado/Rechazado),
                       fecha_pago.
    OrderItem.php    — Ítem de pedido. belongsTo Order y Dish.

  CONTROLADORES (app/Http/Controllers/)
    HomeController.php           — Página principal y menú público
    OrderController.php          — Flujo de pedido del cliente
    KitchenController.php        — Vista y acciones del personal de cocina
    DeliveryController.php       — Vista y acciones del repartidor
    ProfileController.php        — Edición de perfil (Breeze)

    Admin/ (solo accesibles por administrador):
      Admin/UserController.php         — CRUD de usuarios
      Admin/RestaurantController.php   — CRUD de restaurantes + gestión de platos
      Admin/DishController.php         — CRUD de platos
      Admin/CategoryController.php     — CRUD de categorías
      Admin/OrderController.php        — Proceso secundario: gestión de pedidos

  RUTAS (routes/web.php)
    Agrupadas por rol con middleware:
      Públicas                          — sin middleware
      ['auth', 'role:admin']            — solo administrador
      ['auth', 'role:client,admin']     — cliente y admin
      ['auth', 'role:kitchen,admin']    — cocina y admin
      ['auth', 'role:delivery,admin']   — repartidor y admin

  MIDDLEWARE PERSONALIZADO (app/Http/Middleware/CheckRole.php)
    Registrado como alias "role" en bootstrap/app.php.
    Verifica que el usuario autenticado tenga el rol requerido.
    Si no cumple, redirige a la página principal.

  VISTAS (resources/views/)
    layouts/app.blade.php      — Layout principal con sidebar. El rol "client"
                                 solo ve Inicio, Menú y Pedidos (Cocina y Delivery
                                 están ocultos con @if(!auth()->user()->isClient())).
    layouts/guest.blade.php    — Layout split para login/registro.
    dashboard.blade.php        — Inicio con estadísticas.
    admin/                     — Todas las vistas de administración.
    orders/                    — Flujo de pedido del cliente. Incluye selección
                                 de método de pago (tarjeta o Yape con QR), resumen
                                 con categoría y plato en el carrito, y boleta de
                                 detalle del pedido.
    kitchen/                   — Vista del personal de cocina.
    delivery/                  — Vista del repartidor.

  MIGRACIONES (database/migrations/)
    Crean las tablas: users, restaurants, categories, dishes, orders, order_items.
    Siguen convención de clases anónimas requerida por Laravel 12.

  SEEDER (database/seeders/DatabaseSeeder.php)
    Crea 4 usuarios (admin, cocina, cliente, repartidor), 4 restaurantes,
    5 categorías, 29 platos distribuidos entre restaurantes, y 1 pedido de prueba.
    Se ejecuta con: php artisan migrate:fresh --seed --force

5. SUBSANACIÓN DE PROBLEMAS ANTERIORES
========================================

Se corrigieron dos errores detectados en la segunda presentación. El primero era
un error 500 al eliminar un usuario con pedidos asociados (fallo de integridad
referencial). El segundo era un método authorize() que no funcionaba por falta
del trait correspondiente en el controlador base.

  Problema detectado en la segunda presentación:
    Al intentar eliminar un usuario que tenía pedidos asociados, el sistema
    lanzaba un error de integridad de base de datos (error 500).

  Solución aplicada (Admin/UserController.php — método destroy):
    Se agregó verificación previa:
      if ($user->orders()->exists()) {
          return back()->with('error', 'No se puede eliminar...');
      }
    Ahora el sistema muestra un mensaje de error amigable en vez de romperse.

  También se corrigió el método authorize() que no estaba disponible:
    Se agregó el trait AuthorizesRequests en el controlador base (Controller.php).


========================================
6. IMPLEMENTACIÓN DE YAPE
========================================

Para implementar Yape se agregaron tres campos en la tabla orders (metodo_pago,
estado_pago, fecha_pago) mediante migraciones. En el controlador del cliente se
valida que el método de pago sea "Card" o "Yape" y se asigna el estado inicial
según el método. En el controlador admin se creó una ruta PATCH separada para
gestionar el estado de pago con lógica automática de cambio de estado del pedido.

¿Qué es Yape?
  Yape es un método de pago digital alternativo a las tarjetas de crédito.
  En este proyecto, el cliente puede elegir Yape al momento de confirmar su pedido.
  El sistema muestra un código QR estático para el pago, y el administrador confirma
  manualmente cuando recibe el pago en su cuenta.

Flujo de pago con Yape:

  1. Cliente elige Yape en el checkout
     ↓
  2. Sistema muestra código QR para escanear (public/images/yape-qr.jpg)
     ↓
  3. Cliente confirma el pedido → estado: pending, estado_pago: Pendiente
     ↓
  4. Administrador recibe el pago en su cuenta Yape
     ↓
  5. Administrador marca el pago como "Pagado" en /admin/orders
     ↓
  6. Pedido pasa automáticamente a estado "Confirmado"

¿Dónde está implementado Yape en el código?

  a) CREACIÓN DE PEDIDOS (app/Http/Controllers/OrderController.php — método store)

     Validación del método de pago:
       'payment_method' => 'required|in:Card,Yape'
       Solo se permite "Card" o "Yape".

     Asignación del estado de pago:
       'estado_pago' => $request->payment_method === 'Yape' ? 'Pendiente' : null
       Yape → estado_pago: "Pendiente". Tarjeta → estado_pago: null.

     Mensaje diferenciado para el usuario:
       Yape:    "¡Pedido realizado! Tu pago con Yape está pendiente de confirmación."
       Tarjeta: "¡Pedido realizado! Pago con tarjeta registrado."

  b) GESTIÓN DE ESTADO DE PAGO (app/Http/Controllers/Admin/OrderController.php)

     Método updatePaymentStatus():
       Permite al admin cambiar el estado de pago de pedidos con Yape.
       Estados posibles: Pendiente → Pagado o Rechazado.

     Lógica:
       estado_pago = 'Pagado'    → pedido pasa a 'confirmed' automáticamente
       estado_pago = 'Rechazado' → pedido pasa a 'cancelled' automáticamente

  c) MODELO DE ÓRDENES (app/Models/Order.php)

     Campos en la tabla 'orders':
       - metodo_pago (string): Card o Yape
       - estado_pago (string): Pendiente, Pagado, Rechazado
       - fecha_pago (timestamp): Fecha en que se procesa el pago

  d) MIGRACIONES (database/migrations/)

     0009_add_payment_fields_to_orders_table.php
       Agrega los campos metodo_pago, estado_pago y fecha_pago.

     0010_add_missing_payment_fields_to_orders_table.php
       Migración de seguridad: agrega campos si no existen.

     0011_remove_comprobante_pago_from_orders_table.php
       Elimina el campo comprobante_pago que existía en una versión anterior.

  e) VISTA DEL CLIENTE (resources/views/orders/show.blade.php)

     Muestra la boleta con los detalles del pedido:
       - Método de pago (Yape o Tarjeta)
       - Estado de pago (Pendiente, Pagado, Rechazado)
       - Fecha de pago
       - Tabla de platos con cantidad, precio unitario y subtotal
       - Total del pedido
     Incluye botón "Imprimir boleta" (window.print() del navegador).

  f) VISTA DEL ADMINISTRADOR (resources/views/admin/orders/index.blade.php)

     Para pedidos con metodo_pago = 'Yape', aparece un selector:
       - Pendiente (estado inicial)
       - Pagado (confirmar que se recibió el pago → pedido pasa a Confirmado)
       - Rechazado (rechazar el pago → pedido se cancela)

  g) VISTA DEL CLIENTE — CARRITO (resources/views/orders/menu.blade.php)

     Al armar el pedido, el resumen del carrito muestra para cada plato:
       - Categoría en rojo (ej. "Hamburguesas")
       - Nombre del plato
       - Precio unitario y subtotal
     Esto se logra con un objeto JavaScript que mapea dish_id → nombre de categoría,
     construido desde Blade al cargar la vista con los platos del restaurante.


========================================
7. PREVENCIÓN DE ERRORES (try-catch)
========================================

Se envolvieron todos los métodos store(), update() y destroy() de los 5
controladores con bloques try-catch. Las verificaciones de integridad referencial
(relaciones activas) se hacen antes del try-catch con un if explícito para dar
mensajes claros. El catch captura cualquier excepción inesperada y usa
session()->flash() via ->with('error', ...) para mostrar el mensaje en pantalla
sin exponer detalles técnicos al usuario.

Todos los métodos de escritura de los 5 controladores están protegidos con try-catch.
Si ocurre cualquier error inesperado (fallo de BD, excepción de Eloquent, etc.),
el sistema captura la excepción y muestra un mensaje en español en vez de un error 500.

Los mensajes de error según controlador:

  RestaurantController:
    - "No se pudo crear el restaurante. Intente nuevamente."
    - "No se pudo actualizar el restaurante. Intente nuevamente."
    - "No se puede eliminar un restaurante que tiene platos asignados."
    - "No se pudo eliminar el restaurante. Intente nuevamente."

  DishController:
    - "No se pudo crear el plato. Intente nuevamente."
    - "No se pudo actualizar el plato. Intente nuevamente."
    - "No se puede eliminar un plato que tiene pedidos asociados."
    - "No se pudo eliminar el plato. Intente nuevamente."

  CategoryController:
    - "No se pudo crear la categoría. Intente nuevamente."
    - "No se pudo actualizar la categoría. Intente nuevamente."
    - "No se puede eliminar una categoría que tiene platos asignados."
    - "No se pudo eliminar la categoría. Intente nuevamente."

  Admin/OrderController:
    - "No se pudo actualizar el estado. Intente nuevamente."
    - "No se pudo actualizar el estado de pago. Intente nuevamente."

  OrderController (cliente):
    - "No se pudo procesar el pedido. Intente nuevamente."

¿Cómo se muestran los errores en pantalla?
  Todos los métodos usan ->with('error', '...') y ->with('success', '...').
  El layout principal (layouts/app.blade.php) tiene bloques @if(session('error'))
  y @if(session('success')) que muestran el mensaje en un cuadro de alerta
  al inicio de cada página.

¿Las verificaciones previas son parte del try-catch?
  No. Las verificaciones de integridad referencial (ej. "tiene platos asignados")
  se ejecutan antes del try-catch para que el error sea claro y controlado.
  El try-catch solo captura errores inesperados del sistema.
